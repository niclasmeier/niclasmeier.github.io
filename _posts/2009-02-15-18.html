Recently I discovered something new about a thing I am using on an every day basis like most of the software engineers. It's about logging. You may ask: "What is so exiting about logging?"
In my opinion logging is one of the underestimated aspects of software development. Everyone uses it, but only very few are using it right.
Logging is the bridge between the development phase and the operational phase in the software life cycle. If you do it right it may provide valuable informations about the inner workings of an application.

Recently I discovered something new about a thing I am using on an every day basis like most of the software engineers. It's about logging. You may ask: "What is so exiting about logging?"
In my opinion logging is one of the underestimated aspects of software development. Everyone uses it, but only very few are using it right.
Logging is the bridge between the development phase and the operational phase in the software life cycle. If you do it right it may provide valuable informations about the inner workings of an application.

I'am using log4j</a> for quite a while. I am not sure when the framework was created by Ceki Gülcü</a>, but log4j was the straight edge for java logging for the recent years. Unfortunately it became very quite around the log4j project recently.

In my new open source project</a> I broke some new grounds. It started with the simple logging for java (slf4j)</a>, which I used for logging. The motivation was clear, I wanted to get rid of a dedicated logger implementation and offer the possibility to integrate the output of jaev</a> logging into the application log files (streams).
The sl4j</a> is not the first "meta" logging library. The commons logging</a> framework was one of the first of these frameworks. At least it's the one I came across most. The sl4j</a> framework is quite new. It offers a similar functionality like the commons logging</a> framework, but slf4j</a> uses a static binding against the executing logging framework. It also supports various logger implementations and offers bridges from i.e. the JUL (java.util.logging) logging into the sl4j</a> world. So you can integrate all logging informations in the sl4j</a> infrastructure.

This is quite handy if you are combining frameworks which rely on a specific logger implementation (i.e. facelets</a> which relies on JUL). When you use the JUL to slf4j bridge you may merge all logging streams and handle the events on you logger implementations. The slf4j</a> framework by the way was created by Ceki Gülcü</a>.

One other word to the logger implementation. Durning my first steps with slf4j</a> I used the log4j</a> implementation. Which works pretty well. But in last weeks I stumbled about the logback</a> project, an offspring of Ceki Gülcü</a> too. The logback</a> project is a native implementation of the slf4j</a> API and it offers a state of the art logger implementation. If you are using log4j</a> you feel familiar in no time. But it also offers quite some interesting new features and continues the evolution which stalled at the log4j</a> project for quite some years. Here are some highlights:

  • Configuration fully in XML via logback.xml</code> and read automatically on logging initialisation.</li>
  • Different configuration files for productive and test environment (logback.xml</code> and logback-test.xml</code>).</li>
  • Configuration via JMX possible.</li>
  • Quite handy reduction of the logger name length.</li>
  • Provides position informations of the caller (class, line, method).</li>
  • Shortened stack traces of exceptions</li>
    </ul>

    It took not very long to convince me to switch my logging implementation from log4j</a> to logback</a> for the jaev</a> project.
